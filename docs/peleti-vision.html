Un sistema que muestra y gestiona
arte hecho a mano

Peleti es una plataforma de dos capas: un sitio público que presenta el catálogo de piezas artesanales en resina, y un panel de administración que permite gestionar todo el contenido sin tocar código.

01

Propósito del sistema

Peleti nació como un sitio vitrina, pero tiene una necesidad real de gestión de contenido: los productos cambian, las imágenes se actualizan, los textos evolucionan con la marca. Hacer eso a través de código es un cuello de botella innecesario.

El objetivo del sistema es separar la presentación de la operación: cualquier persona con acceso al panel puede agregar piezas al catálogo, actualizar textos del hero, cambiar imágenes del about, o responder mensajes de contacto, todo sin depender de un desarrollador.

Principio rector: el sitio público es la vidriera; el administrador es el depósito y la caja registradora. Ambos hablan el mismo idioma: la misma API, la misma base de datos.

03

Secciones del sitio público

El sitio es una single-page app de scroll continuo. La navegación ancla a cada sección. El orden está diseñado para contar una historia: primero impacto visual, luego contexto, proceso, productos y finalmente conversión.

#inicio

Hero

Imagen de fondo con animación Ken Burns, titular principal, descripción corta y dos CTAs: catálogo y WhatsApp.

#sobre-nosotros

About

Historia de la marca, imagen representativa, filosofía de trabajo y valores artesanales.

#estilos

Galería de Estilos

3 líneas artísticas con técnicas, ejemplos de aplicación e iconografía. Cards con hover reveal.

#proceso

Work Process

Timeline de 5 pasos del proceso de creación: Diseño → Mezcla → Vaciado → Pulido → Control de calidad.

#catalogo

Portafolio / Catálogo

Grilla masonry filtrable por categoría. Modal con imagen grande, dimensiones, técnica y descripción de cada pieza.

#contacto

Contacto

Formulario de email directo + botones a Instagram, Facebook y WhatsApp para conversión inmediata.

Entidades principales (esquema Prisma)

El esquema cubre todas las secciones del sitio. Cada entidad tiene su propio conjunto de endpoints REST con operaciones CRUD completas.

Entidad Qué almacena Endpoint base
Hero Título, descripción, imagen de fondo, textos de botones /api/hero
About Historia de la marca, imagen, párrafos de contenido /api/about
Portfolio Items Nombre, descripción, imagen, dimensiones, técnica, categoría /api/portfolio/items
Portfolio Categories Categorías de piezas (Figurativo, Histórico, Religioso…) /api/portfolio/categories
Styles / StyleGallery Estilos artísticos, técnicas, ejemplos de aplicación /api/styles
Work Process Steps Pasos del proceso creativo con iconos y descripciones /api/work-process/steps
Contact Settings Email, redes sociales, links de WhatsApp /api/contact/settings
Contact Messages Mensajes recibidos desde el formulario público /api/contact/messages
Users Usuario administrador con hash de contraseña /api/users

08

Convenciones del proyecto

Ramas: main es producción estable. develop integra todo el trabajo activo. Features nuevas salen desde develop con el prefijo feature/.

API: todos los endpoints viven bajo /api/ como Route Handlers de Next.js. Los métodos destructivos (POST, PUT, DELETE) requieren header Authorization: Bearer <token>.

Imágenes: se almacenan en /public/images/ y se sirven a través de endpoints dedicados (/api/hero/:id/image) para permitir actualización desde el admin sin cambiar URLs en el frontend.

Datos de contenido: los archivos JSON en src/data/ son la fuente de verdad hasta que el panel admin migre ese contenido a la base de datos. Una vez migrado, los componentes consultan la API en vez de importar JSON.
